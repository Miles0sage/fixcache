## Confirmed Kill Shots

### KS-1: AttributeError cross-attribute false positive (fp7)
**Severity: HIGH — directly violates 100% purity claim**

```
AttributeError: 'NoneType' object has no attribute 'send'
AttributeError: 'NoneType' object has no attribute 'execute'
```

Both hash identically after `_TARGETED_REDACTORS` collapses both to:
`'<type>' object has no attribute '<attr>'`

A user who stores "fix: initialize self.client before calling .send()" will have
that fix retrieved for ANY AttributeError — including ones where `.execute` means
"initialize your DB connection". These are completely different fixes. This is
an intentional design trade-off (cross-repo collapse) but it directly breaks the
100% purity guarantee on a common real-world error class.

**Corpus coverage:** Classes `py-attribute-none` (10 entries in corpus_v2.jsonl)
would all collapse to the same fingerprint.

### KS-2: Unquoted Bearer/JWT token leaks into essence (pl2)
**Severity: HIGH — privacy regression**

```
AuthError: invalid Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.payload.signature
```

The full JWT token survives into `fp.essence` verbatim. The `_TARGETED_REDACTORS`
cover specific prefixes (`sk-`, `ghp_`, `AIza`, `AKIA`, etc.) and `_QUOTED_SINGLE/DOUBLE`
covers quoted values. But unquoted Bearer tokens with no known prefix bypass all
redaction. JWT tokens are extremely common in auth error logs.

**The Darwin export corpus would contain raw JWTs.** Anyone with access to the corpus
could extract valid auth tokens from failed agent runs.

---

### KS-3: Short secrets (< 8 chars) in quotes not redacted (pl4)
**Severity: MEDIUM — documented gap, confirmed real**

```
AuthError: invalid token 'k3y!X9' rejected by server
```

`_QUOTED_SINGLE = re.compile(r"'([^'<>]{8,})'")` — the `{8,}` minimum means any
quoted string under 8 characters is never redacted. API tokens, short passwords,
PINs, 2FA codes, and numeric IDs in quotes all pass through. This was already
documented as `xfail` in `test_fingerprint_hardening.py` but this test confirms
the concrete leakage scenario against a representative input.

## Is the 100% Purity Claim Safe?

**No — with one critical caveat.**

The purity claim holds for **different error classes** (ModuleNotFoundError vs
TypeError vs FileNotFoundError) when ecosystem cues are present. The ecosystem +
error_type combination in the hash key is robust against cross-ecosystem misfires.

The claim breaks for **same-class, same-ecosystem errors with different root causes**:
- All `AttributeError: 'X' object has no attribute 'Y'` collapse to one hash
  regardless of what X or Y are. If a user stores fix A for `.send` and fix B
  for `.execute`, only one fix is retrieved for both — the most recently stored.
- Similarly: all `TypeError: X is not a function` (Node, no stack cue), all
  `Cannot find module '<mod>'`, all `undefined: <name>` (Go) collapse within class.

**The collapse is by design** (cross-repo generalization), but it means:
- The fingerprint is a *class-level* key, not an *instance-level* key
- "100% purity" means "never fires across classes" — which holds
- It does NOT mean "always returns the right fix within a class" — which can fail

For the Darwin dataset, this means the purity guarantee should be scoped as:
**"Zero cross-class misfires"**, not **"zero misfires"**.

## Recommendations

| Priority | Fix |
|---|---|
| HIGH | Add `Bearer\s+\S{20,}` to `_TARGETED_REDACTORS` to catch unquoted JWT/Bearer tokens |
| HIGH | Lower `_QUOTED_SINGLE`/`_QUOTED_DOUBLE` threshold from `{8,}` to `{4,}` |
| MEDIUM | Document purity claim precisely: cross-class only, not within-class |
| LOW | Add `Authorization:\s+\S+` generic redactor for header-logged tokens |
